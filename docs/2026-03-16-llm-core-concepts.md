## 1. RNN의 한계

### 동작 방식
- 텍스트를 **왼쪽에서 오른쪽으로 한 단어씩 순서대로** 처리
- `hidden state`라는 기억 공간에 누적 저장 → 새 단어가 들어올 때마다 **덮어쓰기**

### 3가지 핵심 문제

| 문제 | 설명 |
|---|---|
| 장기 의존성 | 문장이 길면 앞 단어 정보가 사라짐 |
| 순차 처리 | 병렬화 불가 → GPU 수천 코어가 놀고 있음 |
| Gradient Vanishing | 학습 오차 신호가 앞쪽 단어까지 도달 못함 |

### LSTM
- 기억 게이트 추가로 장기 의존성 일부 해결
- 순차 처리 문제는 그대로 → 근본 해결 아님

## 2. Transformer 등장 (2017)

> 논문: **"Attention Is All You Need"** — Google

### 핵심 발상
- RNN을 완전히 버리고 **Attention만으로 문장 전체를 한 번에 병렬 처리**

```java
// RNN 방식 — 순서대로 (느림)
for (String word : sentence) {
    process(word); // 앞 결과 나와야 다음 처리 가능
}

// Transformer 방식 — 병렬 처리 (빠름)
sentence.parallelStream()
        .map(word -> attention(word, allWords))
        .collect(...);
```

### 구조
- **Encoder**: 입력 의미 이해 (Self-Attention + FFN)
- **Decoder**: 출력 생성 (Masked Self-Attention + Cross-Attention)
- GPT/Claude 계열 = **Decoder만 사용**
- BERT 계열 = **Encoder만 사용**

## 3. Attention

### 한 줄 정의
> 현재 단어를 이해하기 위해 문장 전체를 조회해서 맥락을 흡수하는 함수

### DB 조회와 비교

| | DB 조회 | Attention |
|---|---|---|
| Key 매칭 | 정확히 일치 | 유사도로 부드럽게 매칭 |
| 결과 | 해당 row 하나 반환 | 모든 Value를 확률 비율로 섞어 반환 |
| 맥락 반영 | 없음 | 문장 전체 맥락이 결과에 녹아듦 |

### Q / K / V 역할

```
Q (Query)  = 내가 찾고 싶은 것 — 현재 처리 중인 단어
K (Key)    = 각 단어의 검색 키 — 모든 단어의 식별자
V (Value)  = 실제 꺼내올 정보 — 각 단어가 가진 내용
```

### 수식

```
Attention(Q, K, V) = softmax(QKᵀ / √d) · V
```

| 수식 요소 | 의미 |
|---|---|
| `QKᵀ` | Query와 모든 Key의 유사도 계산 (내적) |
| `/ √d` | 값 폭발 방지 스케일 조정 (d = 벡터 차원 수) |
| `softmax` | 유사도를 0~1 확률로 변환 (합계 = 1) |
| `· V` | 확률 가중치로 Value 합산 → 최종 맥락 벡터 |

### 동작 방식
- 단어 3개 문장 → 각 단어가 Query가 되어 전체 문장 조회 → 3×3 행렬 연산으로 **동시 처리**
- 문장이 길수록 연산량이 제곱으로 증가 → RAG 청크 사이즈 고민의 출발점

### Before / After

```
Before: "해외" = [0.8, 0.7]       ← 단어 자체 의미만
After:  "해외" = [0.763, 0.611]   ← 거래·시간대 맥락까지 반영
```

## 4. Multi-Head Attention

### 한 줄 정의
> Attention을 여러 개 병렬로 돌려서 다양한 관점을 동시에 포착하는 구조

### Attention vs Multi-Head Attention

| 비교 항목 | Attention | Multi-Head Attention |
|---|---|---|
| 관점 수 | 1개 | 8~96개 (모델마다 다름) |
| 포착 가능 패턴 | 단일 관계 | 복합 조합 관계 |
| 역할 분담 | 없음 | 학습으로 자동 생성 |
| 연산량 | 1x | Head 수 × 1x (병렬이라 시간 비슷) |

### 핵심 포인트
- Head가 "금액 담당", "위치 담당"으로 나뉘는 건 **사람이 지정하는 게 아님**
- 학습 과정에서 데이터가 역할 분담을 자동으로 만들어냄

### FDS 적용 시나리오

```
Head 1 → 금액 관점 (평소보다 10배)
Head 2 → 위치 관점 (해외 첫 거래)
Head 3 → 시간 관점 (새벽 3시)
Head 4 → 기기 관점 (신규 기기)
         ↓
Concat + Linear → 종합 판단
```

## 5. Embedding

### 한 줄 정의
> 텍스트를 LLM이 처리할 수 있는 숫자 벡터로 변환하는 전처리 게이트웨이

### 왜 단순 숫자 매핑이 안 되나
```
거래=1, 시간대=2, 해외=3 으로 하면
→ 모델이 "해외(3)는 거래(1)보다 3배 크다"로 읽음
→ 의미 없는 크기 관계 생성
```

### 벡터 공간의 특성
- **의미가 비슷한 단어 = 벡터 거리 가까움**

```
"해외" ↔ "국외" ↔ "외국"  → 가까움
"해외" ↔ "사과"            → 멈

왕 - 남자 + 여자 = 여왕  (벡터 연산으로 의미 관계 표현)
```

### RAG와의 연결

```
FDS 점검 보고서
      ↓
nomic-embed-text (Embedding 모델)
      ↓
768차원 벡터
      ↓
Qdrant에 저장

질문 입력 → 동일 모델로 벡터화 → Qdrant 유사도 검색 (= QKᵀ 내적)
```

### 새 단어/개념 학습 방법 3가지

| 방법 | 비용 | 속도 | 새 정보 반영 |
|---|---|---|---|
| Pre-training | 수백억 | 수개월 | 완전 내재화 |
| Fine-tuning | 수십만~억 | 수일~주 | 도메인 특화 |
| RAG | 거의 0 | 실시간 | 즉시 반영 |

> **RAG = 오픈북 시험** — 책(Qdrant)에서 관련 페이지 찾아서 보면서 답변 작성

## 6. Prompt Engineering

### 핵심 원리
> LLM은 확률 예측 기계 → 프롬프트는 그 확률 분포를 원하는 방향으로 조작하는 도구

```
프롬프트가 구체적일수록 → 확률 분포가 원하는 방향으로 집중
프롬프트가 모호할수록   → 확률이 사방으로 퍼짐
```

### 핵심 기법 4가지

**1. Role (역할 부여)**
```
"너는 10년 경력의 금융 보안 전문가야."
→ 해당 역할 벡터 공간이 활성화 → 전문적 어휘 확률 상승
```

**2. Few-shot (예시 제공)**
```
예시 1) 입력: 새벽 3시, 해외, 500만원 → 출력: 고위험
예시 2) 입력: 오후 2시, 국내, 3만원   → 출력: 정상
실제:   입력: 새벽 1시, 국내, 200만원
```
→ 모델이 예시를 Attention으로 참조해서 출력 패턴 맞춤

**3. Chain of Thought (단계적 사고)**
```
"단계별로 분석해줘.
 1단계: 시간대 분석
 2단계: 위치 분석
 3단계: 금액 분석
 4단계: 종합 판단"
```
→ 중간 추론 과정이 다음 토큰의 컨텍스트가 됨 → 최종 답변 품질 향상

**4. Output Format (출력 형식 지정)**
```json
{
  "risk_level": "HIGH/MEDIUM/LOW",
  "reasons": ["이유1", "이유2"],
  "score": 0
}
```
→ Spring에서 LLM 응답 파싱할 때 필수

### FDS AI Assistant 프롬프트 종합 예시

```
[Role]
너는 금융 이상거래 탐지 전문가야.
10년간 국내 주요 은행 FDS를 운영한 경험이 있어.

[Context]
고객 평소 패턴: 국내, 평일 오전, 5~50만원, 동일 기기

[Input]
현재 거래: 새벽 2시, 미국, 300만원, 신규 기기

[Chain of Thought]
1. 시간대 이상 여부
2. 위치 이상 여부
3. 금액 이상 여부
4. 기기 이상 여부
5. 종합 위험도 판단

[Output Format]
{
  "risk_level": "HIGH",
  "triggered_rules": [],
  "ai_score": 0,
  "recommendation": "차단/모니터링/정상"
}
```

## 전체 개념 연결

```
Embedding      →  텍스트를 벡터로 변환 (전처리 게이트웨이)
Attention      →  벡터들이 서로 맥락 흡수 (Q/K/V 조회)
Multi-Head     →  여러 관점에서 동시에 Attention
Prompt Eng.    →  확률 분포를 원하는 방향으로 조작
RAG            →  외부 DB(Qdrant)로 실시간 지식 주입
Fine-tuning    →  벡터 공간 자체를 도메인에 맞게 미세 조정
```

## 하이브리드 FDS 아이디어 (오늘 떠올린 것)

```
소액 거래 (10만원 이하)  →  룰 기반 FDS (비용 0)
고액 거래 (100만원 이상) →  룰 기반 1차 필터
                          →  Multi-Head Attention 2차 정밀 분석
                             (금액·위치·시간·기기 동시 탐지)
```

- 룰 기반: 사람이 정의한 명시적 패턴
- AI 기반: 데이터가 스스로 발견한 복합 패턴 (Impossible Travel 등)
- 비용 효율 + 정확도 동시 확보
